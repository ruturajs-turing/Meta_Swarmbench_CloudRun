# Cloud Terminal Architecture Plan

This is the target design for AegisRun Foundry. The trainer-facing product is not the admin web app and not a browser form for TOML submission. Trainers receive a terminal endpoint. They authenticate, land inside their own parent cloud sandbox, copy or sync task files into that workspace, and run tasks through `runnerctl`. The internal admin panel is only for platform operators to manage users, parent sandboxes, execution sandboxes, costs, templates, failures, and cleanup.

## Product Split

| Surface | Audience | Purpose | Must expose |
| --- | --- | --- | --- |
| Trainer terminal endpoint | Trainers | Work inside a managed parent sandbox and run model-log tasks | SSH/Web terminal, `runnerctl`, task sync, live logs, run status, artifacts |
| Parent sandbox | One per trainer/session | Persistent workspace for task staging and terminal UX | Harbor/task tooling, workspace files, CLI, sync agent, log viewer |
| Execution sandbox | One per task run | Isolated cloud compute created from TOML | CPU/RAM/disk profile, model-log command, log streaming, artifact upload, auto-destroy |
| Control plane | Platform backend | Authority for auth, quotas, provider APIs, secrets, costs, cleanup | APIs, queue, provider adapters, DB, object storage |
| Admin panel | Internal admins only | Manage the whole fleet and all runs | users, sandboxes, runs, costs, templates, kill/pause/refresh, audit logs |

## Correct Mental Model

```text
Trainer local machine
  -> SSH/Web terminal endpoint
  -> Access gateway authenticates username/password or short-lived token
  -> Control plane creates/resumes the trainer's parent cloud sandbox
  -> Trainer lands in /workspace inside the parent sandbox
  -> Trainer runs runnerctl push/run/status/logs/artifacts
  -> Control plane validates TOML and quota
  -> Provider adapter creates one execution sandbox for the run
  -> Execution sandbox runs model logs
  -> Logs/metrics/artifacts stream back to durable storage and parent workspace
  -> Execution sandbox is destroyed
  -> Parent sandbox remains available, pauses when idle, refreshes within 24h
```

The parent sandbox is a workspace, not the owner of the platform. It must not hold provider admin keys or freely create child sandboxes without the control plane. The control plane owns provider credentials, user quotas, billing records, template rollout, run finalization, cleanup, and admin operations.

## Cloud Provider Fit

| Provider | Good fit | Caveat |
| --- | --- | --- |
| E2B | Programmatic sandboxes, custom templates, SSH over WebSocket, command execution, metrics, pause/resume/kill lifecycle | Higher CPU/RAM profiles and Docker-in-sandbox should be validated against account limits before promising 12-16 CPU runs |
| Daytona | SSH-first workflow, token-based SSH access, snapshots, custom resources, web terminal and IDE flows | Default documented org limit is 4 vCPU, 8 GiB RAM, 10 GiB disk, so 12-15 CPU runs need an upgraded/custom arrangement or another provider |
| Self-hosted future | Kubernetes/Firecracker/Kata/gVisor, full resource control, GPUs, lower unit economics | More operational burden, security hardening, networking, billing, and cleanup responsibility |

Use a provider-neutral adapter interface from day one. Do not make the product depend on E2B-only or Daytona-only semantics.

## Source-Backed Provider Notes

E2B templates can define base images, environment variables, files, commands, and start commands, then create sandboxes from the built template. E2B documents sandbox pause, resume/connect, kill, auto-pause behavior, and metrics for CPU, memory, and disk collected every five seconds. E2B also documents SSH access through a WebSocket proxy, which supports remote terminal sessions and SCP/SFTP style transfer.

Daytona documents SSH access using token-based authentication from terminals, IDEs, and dev tools. Daytona sandboxes are isolated environments with dedicated kernel, filesystem, network stack, and allocated resources; snapshots can capture configured environments. The documented default resource ceiling is 4 vCPU, 8 GiB RAM, and 10 GiB disk, and container sandboxes support stop but not pause/resume memory preservation; VM sandboxes support pause/resume.

Sources:

- E2B templates: `https://e2b.dev/docs/template/quickstart`
- E2B persistence/lifecycle: `https://e2b.dev/docs/sandbox/persistence`
- E2B metrics: `https://e2b.dev/docs/sandbox/metrics`
- E2B SSH: `https://e2b.dev/docs/sandbox/ssh-access`
- Daytona SSH: `https://www.daytona.io/docs/en/ssh-access/`
- Daytona sandboxes/resources/lifecycle: `https://www.daytona.io/docs/en/sandboxes/`

## Trainer Terminal Experience

### Entry

Trainer receives one of these:

```bash
ssh trainer_124@gateway.aegisrun.internal
```

or a short-lived command generated by the control plane:

```bash
runnerctl login --endpoint https://gateway.aegisrun.internal
runnerctl ssh
```

The gateway handles authentication, maps the trainer identity to a parent sandbox, creates or resumes that sandbox, and attaches the terminal.

### Parent Sandbox Contents

```text
/opt/aegisrun/
  runnerctl
  terminal-agent
  sync-agent
  log-viewer
  diagnostics
  config/

/workspace/
  tasks/
  runs/
  artifacts/
  cache/
  tmp/

/harbor/
  patched Harbor/tooling or prebuilt runner assets
```

The parent sandbox should include:

| Component | Purpose |
| --- | --- |
| `runnerctl` | Main trainer CLI |
| `terminal-agent` | Heartbeat and session registration |
| `sync-agent` | Upload/download helper for local task folders |
| `log-viewer` | Tails live run logs from durable store/control plane |
| Harbor/task tooling | Maintained by platform template updates |
| Diagnostics | Disk, auth, provider, environment checks |
| No provider admin keys | Parent compromise should not expose fleet credentials |

### Required `runnerctl` Commands

| Command | Purpose |
| --- | --- |
| `runnerctl whoami` | Show user/org/quota |
| `runnerctl workspace` | Show parent sandbox ID, age, disk, template, idle timer |
| `runnerctl doctor` | Validate Harbor/tooling/provider reachability |
| `runnerctl push ./task` | Package and upload local task to parent workspace or object store |
| `runnerctl run ./task/task.toml` | Submit TOML to control plane and start one execution sandbox |
| `runnerctl run ./task --tail` | Submit and follow logs |
| `runnerctl runs` | List active, queued, failed, completed runs |
| `runnerctl status <run_id>` | State, elapsed time, CPU/RAM/disk, cost estimate |
| `runnerctl logs <run_id> --follow` | Stream model logs |
| `runnerctl artifacts <run_id>` | List produced files |
| `runnerctl download <run_id>` | Pull logs/artifacts to parent workspace or local machine |
| `runnerctl stop <run_id>` | Cancel run and destroy execution sandbox |
| `runnerctl retry <run_id>` | New attempt using same normalized TOML |
| `runnerctl clean --workspace` | Clean parent workspace cache/temp files |
| `runnerctl refresh-parent` | Request parent sandbox refresh |
| `runnerctl logout` | End terminal token/session |
| `runnerctl tui` | Menu-driven terminal UI for trainers who should not memorize commands |

### Menu-Driven Terminal UI

The default trainer experience should be a guided terminal screen after login. It should show:

- current endpoint and logged-in user;
- parent sandbox ID, provider, template version, and state;
- active execution count against quota;
- recent runs with state and cost;
- numbered actions for upload, run, logs, artifacts, download, stop, retry, and SSH token.

The menu mode is intentionally command-backed. Every menu action should call the same `runnerctl` API paths as the direct commands, so power users can script the CLI and new trainers can use the guided flow.

## Task Copying Modes

This is a major product caveat: a remote SSH parent sandbox cannot magically read a trainer's local files. Support these modes explicitly.

| Mode | Trainer command | Best for | Notes |
| --- | --- | --- | --- |
| Local CLI push | `runnerctl push ./task && runnerctl run task.toml` from trainer machine | Best default workflow | Requires trainer to install small local CLI |
| Rsync/SFTP | `rsync -av ./task trainer@gateway:/workspace/tasks/task-a/` | Power users | Simple and transparent |
| Remote-first | `git clone ...` inside parent sandbox | Repo-based tasks | Avoids local upload |
| Web upload fallback | Admin/trainer portal upload to object store | Non-technical users | Not primary interface |
| IDE remote | VS Code/JetBrains over SSH | Long editing sessions | Parent sandbox becomes remote dev box |

Recommended MVP: local CLI push plus SSH/rsync fallback. Do not rely only on browser upload.

## Execution Sandbox Lifecycle

Execution sandboxes are single-use. A run is not a long-lived container inside the parent unless a provider explicitly makes that safe and cheap. The control plane should prefer a provider-created execution sandbox per run.

```text
REQUESTED
  -> VALIDATING_TOML
  -> QUOTA_CHECKED
  -> QUEUED
  -> PROVISIONING_EXECUTION_SANDBOX
  -> STAGING_INPUTS
  -> RUNNING_MODEL_LOGS
  -> STREAMING_LOGS_AND_METRICS
  -> COLLECTING_ARTIFACTS
  -> FINALIZING_RESULT
  -> SUCCEEDED / FAILED / TIMED_OUT / CANCELLED / INFRA_FAILED
  -> DESTROYING_EXECUTION_SANDBOX
  -> DESTROYED
```

The execution sandbox receives:

- normalized TOML;
- task bundle or object-store pointer;
- allowed image/template;
- resource profile mapped to provider limits;
- short-lived secrets;
- run-agent binary;
- output manifest rules.

The execution sandbox must return:

- stdout/stderr logs;
- model-log files;
- verifier output;
- result summary;
- artifacts archive;
- resource metrics;
- final state and exit code.

## TOML Resource Handling

The TOML should not directly get unlimited power. It requests a profile, and the control plane maps that to provider capabilities.

| Requested profile | Target resources | Provider handling |
| --- | ---: | --- |
| `small` | 2 vCPU, 4 GiB RAM, 20 GiB disk | Daytona default/custom or E2B template |
| `medium` | 4 vCPU, 8 GiB RAM, 40 GiB disk | Daytona upper default ceiling or E2B |
| `large` | 8 vCPU, 16 GiB RAM, 80 GiB disk | E2B Pro/Enterprise or custom provider |
| `xlarge` | 12-16 vCPU, 24-32 GiB RAM, 100+ GiB disk | Custom E2B/Daytona contract or self-hosted adapter |

Validation rules:

- reject raw unbounded CPU/RAM/disk;
- require image/template allowlist;
- require digest-pinned images for production;
- require timeout and max output size;
- default network deny or domain allowlist;
- inject secrets by reference only;
- enforce two active execution sandboxes per trainer unless admin overrides.

## Parent Sandbox Lifecycle

```text
NONE
  -> CREATING
  -> READY
  -> CONNECTED
  -> IDLE
  -> PAUSING_OR_STOPPING
  -> PAUSED_OR_STOPPED
  -> RESUMING
  -> REFRESH_PENDING
  -> DRAINING
  -> RECREATED_FROM_TEMPLATE
  -> DESTROYED
```

Rules:

- one active parent sandbox per trainer by default;
- no new runs when parent is `REFRESH_PENDING`;
- active execution runs continue even if parent disconnects;
- parent refresh every 24 hours through drain/recreate, not blind deletion;
- idle parent auto-pauses or stops depending on provider capability;
- parent disk cleanup runs before refresh;
- workspace metadata syncs to object storage before destroy/recreate.

## Admin Panel Scope

The admin panel is for internal operators only. It should not be the trainer product.

### Admin Dashboard

| View | Shows | Actions |
| --- | --- | --- |
| Fleet | active parents, active executions, queue depth, provider health, cost/hour | provider disable, global kill switch, cleanup scan |
| Users | trainer accounts, parent sandbox, active runs, quota, spend | create/disable user, reset password, set quota |
| Parent sandbox detail | provider ID, template version, age, disk, idle timer, SSH sessions | pause/stop/resume, refresh, destroy, open logs |
| Run detail | TOML, execution sandbox ID, logs, metrics, artifacts, result, cost | stop, retry, force cleanup, download bundle |
| Templates | parent template versions, Harbor patch version, runner image digest | canary rollout, promote, rollback, force refresh |
| Cost | per-user, per-run, provider cost, storage, egress, orphan spend | budgets, alerts, caps |
| Cleanup center | orphan sandboxes, stuck states, failed artifact uploads | reconcile, kill, mark infra failed |
| Audit log | admin actions, user sessions, secret access, run cancellation | export, filter |

### Admin Actions

- create trainer login;
- issue/revoke terminal tokens;
- view every parent sandbox and execution sandbox;
- force stop run;
- destroy orphan execution sandbox;
- pause/stop/resume parent sandbox;
- refresh parent sandbox from latest template;
- set active-run limit, e.g. two simultaneous runs;
- set monthly budget and max runtime;
- promote parent sandbox template;
- promote runner image/Harbor patch;
- disable provider;
- emergency kill all new runs.

## Update And Patch Strategy

Parent sandboxes must be maintained centrally.

```text
Git change
  -> CI builds parent template
  -> install Harbor patches, runnerctl, terminal-agent, sync-agent
  -> vulnerability scan
  -> create provider template/snapshot
  -> canary to internal test user
  -> promote template version
  -> mark existing parents refresh_pending
  -> refresh idle parents
  -> drain and refresh active parents within 24h
```

Do not patch live parent sandboxes manually except for emergency debugging. Live manual changes will drift and become impossible to support.

## Cost Controls

| Control | Rule |
| --- | --- |
| Active execution runs | Default two per trainer |
| Parent lifetime | Refresh at least every 24h |
| Parent idle | Pause/stop after idle timeout |
| Execution sandbox TTL | Hard timeout from TOML and quota |
| Artifact size | Reject or truncate over max output |
| Provider budget | Stop new runs when budget threshold is reached |
| Cleanup finalizer | Always destroy execution sandbox after finalization |
| Reconciler | Scan provider inventory for leaked sandboxes |
| Admin kill switch | Disable new runs globally or per provider |

## Implementation Phases

### Phase 1 - Terminal MVP

- Control plane with users, sessions, quotas, parent sandboxes, runs.
- One provider adapter, preferably the one that proves SSH fastest.
- Parent sandbox template with `runnerctl`, Harbor/tooling, sync-agent.
- Gateway that maps trainer login to parent sandbox.
- `runnerctl whoami`, `workspace`, `push`, `run`, `status`, `logs`, `artifacts`, `stop`.
- One execution sandbox per run.
- Logs and artifacts persisted outside sandboxes.
- Admin fleet view and run detail.

### Phase 2 - Admin Operations

- Parent sandbox inventory and actions.
- Execution sandbox inventory and force cleanup.
- Template version management.
- User quota editor.
- Cost dashboard.
- Audit log.
- Cleanup reconciler.

### Phase 3 - Production Hardening

- Short-lived SSH tokens.
- Secret vault integration.
- Provider failover.
- Resource profile mapping.
- Network allowlists.
- Object-store signed download URLs.
- Template canary/rollback.
- Budget alerts.
- Orphan sandbox sweeper.

### Phase 4 - Multi-Provider

- E2B adapter.
- Daytona adapter.
- Optional self-hosted adapter.
- Provider selection policy by resource profile, cost, availability, and SSH support.
- Cross-provider cost/runtime reporting.

## Key Caveats

- A parent sandbox cannot read local trainer files unless the trainer uploads/syncs them or uses a local CLI.
- Daytona default documented limits do not support 12-15 CPU execution sandboxes without custom limits.
- E2B supports useful lifecycle controls, but high resource profiles and Docker-in-sandbox must be validated on the actual account.
- Container pause/resume behavior differs by provider and sandbox class; design for provider capability flags.
- Provider SSH support should be hidden behind your own gateway/session model so trainer UX does not change if provider changes.
- Raw provider API keys must never enter parent sandboxes.
- Execution sandbox cleanup must be finalizer plus reconciler, because finalizers fail.
- Parent refresh must drain active work; do not delete a parent while an upload or terminal session is active.
